# Agent Collaboration

## Active Agents

- Claude Sonnet 5 (Claude Code) — user reported the engine dropping to ~900 FPS on an RX 9070 for a simple scene with bloom + basic lighting, way below what that GPU should need. Investigated and fixed two real CPU-side per-frame inefficiencies (not GPU work — a scene this simple should be GPU-trivial on that hardware). Uncommitted as of this writing.

## What was found and fixed

1. **Descriptor pool churn (the big one)**: `Core/src/Core/Vulkan/Rhi/VulkanRhiBridgeBinding.cpp`'s `create_bind_group()` created a brand-new dedicated `VkDescriptorPool` for *every single bind group*, every call — the code's own old comment even said so: "Each bind group owns a private pool sized to exactly its one set... a shared growable pool allocator is a later optimization once bind-group churn matters." It now matters: this renderer creates ~10-15+ bind groups per frame (deferred lighting, every bloom mip level, the new bloom composite pass, custom post-processes, tonemap), so this was potentially tens of thousands of `vkCreateDescriptorPool`/`vkDestroyDescriptorPool` calls per second — a well-known, expensive Vulkan anti-pattern.
   - **Fix**: added a shared, growable pool of `DescriptorPoolChunk`s (`VulkanRhiBridge.hpp`/`VulkanRhiBridgeBinding.cpp`) that ordinary bind groups suballocate from — sized generously (256 sets, 1024 descriptors/type), grown (a new chunk appended) only when the current one's allocation fails, individual sets freed back via `vkFreeDescriptorSets` on `destroy_bind_group`. Update-after-bind/variable-descriptor-count/acceleration-structure bind groups (rare, bindless-shaped) keep the old dedicated-pool-per-set path unchanged — not worth the complexity of sharing chunks for those.
   - Guarded with `Async::Mutex<vector<DescriptorPoolChunk>>` (not a bare `std::mutex`) since `create_bind_group`/`destroy_bind_group` are documented as safe to call concurrently across windows, and Vulkan requires external synchronization per `VkDescriptorPool`.
   - Added a one-time `Foundation::log_info` when a new chunk is created (a legitimate permanent diagnostic, matching this file's other cold-start-cost logs) — verified empirically: over a 15-second run at whatever framerate this produced, exactly **one** chunk was ever created, vs. what would have been many thousands of individual pool creates under the old code.
2. **Bloom composite target recreated every frame**: the render graph is rebuilt from scratch every frame by design (a fresh `RenderGraph graph;` local each `render_frame_rhi()` call), which is fine for the *declarations*, but the new bloom-composite pass's output was a `graph.create_texture()` (transient) — meaning a fresh `VkImage`+`VkImageView` (and VMA allocation) got minted and torn down *every single frame*, for what is the exact same logical resource (same extent/format) every frame bloom is active — which is the default, so this hit every frame of the exact scene the user measured.
   - **Fix**: added `FrameCompositeTarget` to `FrameInFlight` (`RendererModule.hpp`) plus `ensure_frame_composite_target()`/`destroy_frame_composite_target()` (`RendererLifecycle.cpp`), mirroring the exact pattern `FrameDeferredTargets`/`FrameBloomTargets` already use — a persistent, resize-on-demand allocation instead of a per-frame transient one. The render graph now `import_texture()`s it (same `initial_layout = Undefined` convention every other persistent frame-slot target already uses, since it's always fully overwritten via `LoadOp::DontCare`) instead of `create_texture()`ing it. Wired into the prewarm section and every teardown/resize site that already tears down bloom targets.

## Verification

- Full clean rebuild (`cmake --build build/x86-64/linux/debug`, all targets) — no errors.
- `EcsSmokeTest` and `RenderGraphTest` (the two CPU-only test suites) both still pass.
- Ran `Runtime` directly (backgrounded, 15s): correct Vulkan/RHI init, stable memory, zero error/validation lines, clean shutdown. Exactly 1 descriptor pool chunk created over the whole run (vs. the old per-bind-group churn).
- Could not get an exact FPS before/after number in this sandboxed session (no way to read the native Wayland window's title bar, which is where Runtime shows live FPS) — the fix is validated by the pool-chunk-count evidence and by it being a well-documented Vulkan anti-pattern the code's own comments already flagged as a known future cost, not by a measured FPS delta. Worth the user checking the in-app FPS counter themselves to confirm the real-world win.

## Not done here (identified, lower priority / bigger scope)

- Custom-post-process *intermediate* targets (when a custom effect is actually configured — off by default in Runtime today) still use `graph.create_texture()` per frame. Same class of fix as the bloom composite target would apply, just not the default hot path, so left alone.
- The render graph's own per-frame CPU bookkeeping (rebuilding `textures_`/`physical_slots_`/`ordered_passes_`/`render_passes_` vectors and ~15 `std::function`-captured pass closures from scratch every frame) is real but much smaller overhead than the two fixes above — not touched. This is generally the kind of cost a persistent, cross-frame frame-graph-object/resource-pool architecture (see the still-open graph-hardening item 2/3 from the earlier joint plan) would eventually amortize away, but there's no evidence yet it's significant enough to justify that scope right now.
